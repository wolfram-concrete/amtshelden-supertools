Kreisläufe (halten alles aktuell & monetarisieren)

Re-Check-Schleife: Scheduler prüft veröffentlichte Tools nach Intervall erneut (Health-Check + Compliance) → bei Abweichung Status „aktualisieren/Problem" → zurück in die Redaktion.
Korrektur-Schleife: „Daten veraltet?"-Widget auf der Website → Meldung landet im Dashboard → Redaktion prüft → Profil aktualisiert.
Lead-Schleife (Erlös): Behörde stellt Anfrage → Routing über Amtshelden → qualifiziert → Anbieter kontaktiert → Status der Anfrage im Dashboard nachverfolgt.

Für die Entwicklung — Annahmen & offene Fragen

  • Stack-Vorschlag: Next.js (vorhanden) + Payload CMS / PostgreSQL (Supabase EU) als SSoT; Crawler als separater Worker (Cron/Queue); Dashboard als geschützter Bereich derselben App.
  • Crawler: wie viel automatisch vs. KI-assistiert? Rechtlicher Rahmen fürs Crawlen (robots, ToS)?
  • confidence/review_status sind schon im Datenmodell angelegt — Dashboard baut darauf auf.
  • Re-Check-Intervall pro Tool/Compliance-Feld definierbar (z. B. DSGVO halbjährlich)?
  • Mensch-Gate bleibt Pflicht vor „veröffentlicht" — kein Auto-Publish (Markenversprechen Kuratierung).
  • Schnittstellen: Crawler→DB, Dashboard→DB, Website liest nur „veröffentlicht", Korrektur-Widget→DB.
  • Offen: Anbieter-Self-Service-Login (Profil pflegen) ab welcher Phase? Eigene Tabelle/Rollen.
  • Empfehlung erbeten: Build-vs-Buy fürs Crawling, Queue/Scheduler-Tech, Auth/Rollen fürs Dashboard.
